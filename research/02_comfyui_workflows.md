## 1. Installation and Setup on Windows

### Option A: ComfyUI Desktop (Recommended)

ComfyUI Desktop is the official one-click installer for Windows. It handles Python environment setup, dependency management, and model configuration automatically.

**System Requirements:**
- NVIDIA GPU with at least 4 GB VRAM (8+ GB recommended, 16-24 GB ideal)
- At least 16 GB system RAM (32 GB recommended for complex workflows)
- SSD with at least 15 GB free space (more needed for models)
- Windows 10/11

**Installation Steps:**
1. Download the installer from: https://download.comfy.org/windows/nsis/x64
2. Double-click the installer; it creates a desktop shortcut automatically.
3. On first launch, select GPU mode:
   - **NVIDIA GPU (Recommended)**: Direct PyTorch + CUDA support
   - Manual Configuration: for advanced users
   - CPU Mode: for development only (very slow)
4. Choose an installation directory on an SSD; create a dedicated folder.
5. Optionally migrate existing ComfyUI installations (models, custom nodes, workflows).
6. Configure update preferences and mirror settings.

**Key File Locations:**
- Logs: `C:\Users\<USERNAME>\AppData\Roaming\ComfyUI\logs`
- Configuration: `C:\Users\<USERNAME>\AppData\Roaming\ComfyUI`

### Option B: Portable / Manual Installation

For programmatic/API use (our use case), the manual install is often better:

```bash
# Clone the repository
git clone https://github.com/comfyanonymous/ComfyUI.git
cd ComfyUI

# Create a Python virtual environment (Python 3.10+ required)
python -m venv venv
venv\Scripts\activate

# Install PyTorch with CUDA support
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu121

# Install ComfyUI dependencies
pip install -r requirements.txt
```

**Model Directory Structure:**
```

    clip/            # CLIP text encoder models

    loras/           # LoRA files

    vae/             # VAE models
    upscale_models/  # Upscaler models
```

```bash

comfy launch -- --listen 0.0.0.0 --port 8188  # With custom options
```

## 2. ComfyUI Workflow Fundamentals

### Workflow Format

ComfyUI workflows are directed acyclic graphs (DAGs) stored as JSON. Each node performs a specific operation, and outputs connect to inputs of downstream nodes.

**Two JSON formats exist:**
1. **Workflow format**: Contains visual layout + node data (used by the UI)
2. **API format**: Stripped-down format for programmatic execution (used by the API)

To export API format: Enable "Dev Mode" in ComfyUI Settings, then click "Save (API Format)".

### Loading Workflows

- Drag and drop a `.json` file onto the ComfyUI interface
- Drag and drop a `.png` image (workflow metadata is embedded in PNG files generated by ComfyUI)
- Use the Load button in the menu

### Basic Text-to-Image Workflow Nodes

A minimal workflow consists of:

```
CheckpointLoaderSimple --> CLIPTextEncode (positive prompt)
                      \--> CLIPTextEncode (negative prompt)
                       \--> EmptyLatentImage
                        \--> KSampler --> VAEDecode --> SaveImage
```

**Node Descriptions:**
| Node | Purpose |
|------|---------|
| `CheckpointLoaderSimple` | Loads the base model (checkpoint) |
| `CLIPTextEncode` | Converts text prompt to conditioning |
| `EmptyLatentImage` | Creates a blank latent at specified resolution |
| `KSampler` | The main diffusion sampling node |
| `VAEDecode` | Converts latent to pixel image |
| `SaveImage` | Saves the output to disk |

## 3. Logo Generation Workflows

### Workflow A: LoRA-Based Logo Generation (Simplest)

This is the easiest approach, using a logo-specific LoRA on top of an SDXL checkpoint.

**Nodes Required:**
```
CheckpointLoaderSimple
  --> LoraLoader (load logo LoRA)
    --> CLIPTextEncode (positive: "logo of [business], vector, minimal, clean")
    --> CLIPTextEncode (negative: "photo, realistic, blurry, text artifacts")
      --> EmptyLatentImage (1024x1024)
        --> KSampler
          --> VAEDecode
            --> SaveImage
```

**Recommended Settings:**
- Resolution: 1024x1024 (square, matching SDXL training)
- Sampler: `dpmpp_2m` or `dpmpp_sde`
- Scheduler: `karras`
- Steps: 25-35
- CFG Scale: 7-8
- Batch size: 4 (for generating multiple variations)

**Key Tips:**
- The LoRA must match the base checkpoint's architecture (SDXL LoRA with SDXL checkpoint)
- Include trigger words specific to the LoRA (e.g., "LogoRedAF" for Logo.Redmond, "logomkrdsxl" for Logo Maker 9000)
- Include "logo" and "vector" in the positive prompt
- Use negative prompt to suppress photorealism and artifacts

### Workflow B: ControlNet-Guided Logo Generation

Use a sketch or draft as a structural guide for the generated logo.

**Three ControlNet Options:**

1. **Canny ControlNet**: Best for drafts from graphics software (Illustrator, Figma). Preserves edge outlines precisely.
2. **Scribble ControlNet**: Best for hand-drawn sketches. More forgiving of rough input.
3. **Depth ControlNet**: Best for logos that need dimensional/layered appearance.

**Nodes Required:**
```
CheckpointLoaderSimple
  --> ControlNetLoader (load canny/scribble model)
    --> CannyEdgePreprocessor (or appropriate preprocessor)
      --> ControlNetApplyAdvanced
        --> CLIPTextEncode (positive)
        --> CLIPTextEncode (negative)
          --> KSampler --> VAEDecode --> SaveImage
```

**Critical Rule:** The ControlNet model's base architecture must match the checkpoint (e.g., SD 1.5 ControlNet with SD 1.5 checkpoint, SDXL ControlNet with SDXL checkpoint).

### Workflow C: Reference Image (IPAdapter + ControlNet)

Use an existing logo or style reference to guide generation.

**Method 1 - Reference-Only ControlNet:**
- Requires the `ComfyUI_experiments` plugin
- Uses the `ReferenceOnlySimple` node
- Converts reference image to latent via VAE Encode
- Works well for logo style transfer

**Method 2 - IPAdapter:**
- Uses `IPAdapterModelLoader` to load IP-Adapter model
- Transfers style from a reference image
- Less effective for logo specifics, better for general style matching
- Best combined with ControlNet for structure

### Workflow D: Flux-Based Logo Generation

See [Section 7](#7-flux-setup-for-logo-generation) for detailed Flux setup.

Flux excels at text rendering in images, making it particularly valuable for logos that incorporate text/wordmarks.

## 5. Running ComfyUI as an API Server

### Starting the Server

**Default (with UI):**
```bash
python main.py
# Starts on http://127.0.0.1:8188
```

**Headless / API mode (accessible from other machines):**
```bash
python main.py --listen 0.0.0.0 --port 8188
```

**With comfy-cli:**
```bash
comfy launch -- --listen 0.0.0.0 --port 8188
```

**Disable the UI preview (reduces overhead):**
```bash
python main.py --listen 0.0.0.0 --port 8188 --preview-method none
```

### Core API Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/prompt` | POST | Queue a workflow for execution |
| `/history/{prompt_id}` | GET | Get execution results/output info |
| `/view` | GET | Retrieve generated images |
| `/upload/image` | POST | Upload input images |
| `/ws?clientId={id}` | WebSocket | Real-time progress tracking |
| `/queue` | GET | View current queue state |
| `/interrupt` | POST | Cancel current generation |

### Exporting API-Format Workflows

1. In ComfyUI, click the Settings gear icon
2. Enable "Dev Mode Options"
3. The "Save (API Format)" button appears in the menu
4. Click it to export the workflow JSON in API-compatible format

The API format differs from the standard workflow format -- it strips out UI layout data and uses node IDs as keys directly.

## 6. Python Integration

### Method 1: Direct HTTP/WebSocket API

This is the primary integration method for production use.

```python
import json
import uuid
import urllib.request
import urllib.parse
import websocket  # pip install websocket-client

SERVER_ADDRESS = "127.0.0.1:8188"
CLIENT_ID = str(uuid.uuid4())


def queue_prompt(prompt: dict) -> dict:
    """Submit a workflow to ComfyUI for execution."""
    payload = {"prompt": prompt, "client_id": CLIENT_ID}
    data = json.dumps(payload).encode("utf-8")
    req = urllib.request.Request(
        f"http://{SERVER_ADDRESS}/prompt", data=data
    )
    return json.loads(urllib.request.urlopen(req).read())


def get_history(prompt_id: str) -> dict:
    """Retrieve execution history/results."""
    with urllib.request.urlopen(
        f"http://{SERVER_ADDRESS}/history/{prompt_id}"
    ) as response:
        return json.loads(response.read())


def get_image(filename: str, subfolder: str, folder_type: str) -> bytes:
    """Download a generated image from the server."""
    params = urllib.parse.urlencode({
        "filename": filename,
        "subfolder": subfolder,
        "type": folder_type,
    })
    with urllib.request.urlopen(
        f"http://{SERVER_ADDRESS}/view?{params}"
    ) as response:
        return response.read()


def generate_and_wait(workflow: dict) -> list:
    """Submit workflow, wait for completion, return image data."""
    ws = websocket.WebSocket()
    ws.connect(f"ws://{SERVER_ADDRESS}/ws?clientId={CLIENT_ID}")

    result = queue_prompt(workflow)
    prompt_id = result["prompt_id"]

    # Wait for execution to complete
    while True:
        msg = ws.recv()
        if isinstance(msg, str):
            data = json.loads(msg)
            if data["type"] == "executing":
                exec_data = data["data"]
                if exec_data["node"] is None and exec_data["prompt_id"] == prompt_id:
                    break  # Execution complete

    ws.close()

    # Retrieve generated images
    history = get_history(prompt_id)
    images = []
    for node_id, node_output in history[prompt_id]["outputs"].items():
        if "images" in node_output:
            for img_info in node_output["images"]:
                img_data = get_image(
                    img_info["filename"],
                    img_info["subfolder"],
                    img_info["type"],
                )
                images.append(img_data)
    return images
```

### Method 2: Modifying Workflow Parameters Dynamically

```python
import json
import random

# Load an API-format workflow
with open("logo_workflow_api.json", "r") as f:
    workflow = json.load(f)

# Modify parameters by node ID
# (Node IDs are the keys in the API-format JSON)
workflow["6"]["inputs"]["text"] = "minimalist logo for a coffee shop, vector, flat design"
workflow["7"]["inputs"]["text"] = "photorealistic, blurry, noisy, complex"

# Submit

```

## 7. Flux Setup for Logo Generation

### Why Flux for Logos?

Flux.1 by Black Forest Labs excels at:
- **Text rendering**: Accurately generates text within images (critical for wordmark logos)
- **Complex compositions**: Handles multi-element designs well
- **Visual quality**: High detail and coherence
- **Prompt adherence**: Strong alignment with descriptive prompts

### Model Variants

| Variant | License | Quality | Speed | Notes |
|---------|---------|---------|-------|-------|
| **Flux.1 Dev** | Non-commercial | Best | Slow (50 steps) | Best for quality logo work |
| **Flux.1 Schnell** | Apache 2.0 | Good | Fast (4 steps) | Good for rapid prototyping |
| **Flux.1 Dev FP8** | Non-commercial | Near-best | Moderate | Lower VRAM, single file |
| **Flux.1 Dev GGUF** | Non-commercial | Good | Moderate | Lowest VRAM (6GB min) |
| **Flux.1 Dev NF4** | Non-commercial | Good | Moderate | Quantized, low VRAM |

### Download URLs and File Placement

#### FP8 Single-File Version (Easiest Setup)

| File | URL | Place In |
|------|-----|----------|
| `flux1-dev-fp8.safetensors` | https://huggingface.co/Comfy-Org/flux1-dev/blob/main/flux1-dev-fp8.safetensors | `models/checkpoints/` |
| `flux1-schnell-fp8.safetensors` | https://huggingface.co/Comfy-Org/flux1-schnell/blob/main/flux1-schnell-fp8.safetensors | `models/checkpoints/` |

With FP8, use a standard `CheckpointLoaderSimple` node -- no special nodes needed.

#### Full-Precision Version (Best Quality)

| File | Size | URL | Place In |
|------|------|-----|----------|
| `flux1-dev.safetensors` | 23.8 GB | https://huggingface.co/black-forest-labs/FLUX.1-dev | `models/unet/` |
| `clip_l.safetensors` | 246 MB | (included in Flux repo) | `models/clip/` |
| `t5xxl_fp16.safetensors` | 9.79 GB | (for 32GB+ VRAM) | `models/clip/` |
| `t5xxl_fp8_e4m3fn.safetensors` | 4.89 GB | (for 8-12GB VRAM) | `models/clip/` |
| `ae.safetensors` | 335 MB | (rename to `flux_ae.safetensors`) | `models/vae/` |

#### GGUF Version (Low VRAM, 6GB minimum)

**Required custom node:** https://github.com/city96/ComfyUI-GGUF

| File | URL | Place In |
|------|-----|----------|
| GGUF UNET model | https://huggingface.co/city96/FLUX.1-dev-gguf | `models/unet/` |
| `t5-v1_1-xxl-encoder-gguf` | (from GGUF repo) | `models/clip/` |
| `clip_l.safetensors` | (same as above) | `models/clip/` |
| `ae.safetensors` | (rename to `flux_ae.safetensors`) | `models/vae/` |

### Flux Workflow Nodes (Full-Precision Version)

```
UNETLoader (flux1-dev.safetensors)
DualCLIPLoader (clip_l.safetensors + t5xxl_fp8_e4m3fn.safetensors)
VAELoader (flux_ae.safetensors)
  --> CLIPTextEncode (positive prompt)
    --> EmptySD3LatentImage (1024x1024)
      --> KSampler
        --> VAEDecode
          --> SaveImage
```

**Critical Flux KSampler Settings:**
- **CFG: 1.0** (Flux uses guidance differently -- do NOT use high CFG values)
- **Sampler**: `euler`
- **Scheduler**: `simple`
- **Steps**: 20-50 for Dev, 4 for Schnell

### Flux Workflow Nodes (FP8 Checkpoint Version)

```
CheckpointLoaderSimple (flux1-dev-fp8.safetensors)
  --> CLIPTextEncode (positive prompt)
    --> EmptySD3LatentImage (1024x1024)
      --> KSampler (cfg=1.0, euler, simple)
        --> VAEDecode
          --> SaveImage
```

### Flux LoRA for Logos

Flux LoRAs use a different loader node and are placed in `models/loras/`. Typical LoRA strength for Flux is 0.8.

```
UNETLoader --> LoraLoader (strength_model=0.8)
DualCLIPLoader --> LoraLoader (strength_clip=0.8)
  --> (rest of workflow)
```

| Model | Purpose |
|-------|---------|

| **RealESRGAN_x4plus** | Good general-purpose upscaler |

Place upscaler models in `models/upscale_models/`.

### LoRA Usage Guidelines

- **Match architectures**: An SDXL LoRA only works with SDXL checkpoints. A Flux LoRA only works with Flux models.
- **LoRA strength**: Start at 0.7-0.8 and adjust. Too high can cause artifacts.
- **Chaining LoRAs**: Multiple `LoraLoader` nodes can be connected in series to combine effects (e.g., logo style + color palette).
- **Trigger words**: Always check the LoRA's Civitai page for required trigger words to include in your prompt.
- **Testing**: Generate at batch_size=4 with different seeds to compare outputs before selecting the best.

## 9. Background Removal and Post-Processing

For logos, transparent backgrounds are essential. Several ComfyUI nodes handle this:

### Option 1: rembg Node

**Install:** `rembg-comfyui-node` via Manager or https://github.com/Jcd1230/rembg-comfyui-node

```
VAEDecode --> Image Rembg (Remove Background) --> SaveImage
```

Uses the `rembg` library with U2Net model for background segmentation.

### Option 2: easy imageRemBg

**Install:** `ComfyUI-Easy-Use` via Manager

Supports multiple removal models:
- **RMBG-1.4 / RMBG-2.0**: Good general-purpose removal
- **Inspyrenet**: High accuracy on complex edges
- **BEN2**: Latest model with strong edge handling

Has a `post_processing` boolean that applies additional refinement filters.

### Option 3: TransparencyBackgroundRemover

**Install:** https://github.com/Limbicnation/ComfyUI-TransparencyBackgroundRemover

Features edge refinement and foreground bias adjustment -- useful for logos with fine details.

### Saving with Transparency

Use the `SaveImage` node's PNG output (which supports alpha channels) or dedicated transparent-save nodes from WAS Node Suite.

### Post-Processing Pipeline for Logos

A typical logo post-processing workflow:

```
VAEDecode
  --> Upscale (4x-UltraSharp)
    --> Background Removal (rembg/RMBG)
      --> SaveImage (PNG with transparency)
```

## 10. Practical Implementation Plan

### Recommended Architecture for logo-gen Project

```
logo-gen/
  comfyui_server/           # ComfyUI installation
  workflows/
    logo_lora_sdxl.json     # LoRA-based SDXL workflow (API format)
    logo_controlnet.json    # ControlNet-guided workflow
    logo_flux_text.json     # Flux workflow for text-heavy logos
  scripts/
    comfyui_client.py       # Python client for ComfyUI API
    generate_logo.py        # High-level logo generation interface
  models/                   # Symlinked or shared model directory
```

### Step-by-Step Setup Plan

1. **Install ComfyUI** (portable version for API use)
2. **Install ComfyUI Manager**
3. **Download models:**
   - Base: `flux1-dev-fp8.safetensors` (for text logos) + SDXL 1.0 (for LoRA logos)
   - LoRAs: `Logo.Redmond`, `Logo Maker 9000`, `Modern Logo v1.1`
   - ControlNet: SDXL Canny ControlNet
   - Background removal: RMBG-2.0
   - Upscaler: 4x-UltraSharp
4. **Install custom nodes:** WAS Node Suite, ComfyUI Essentials, rembg node, ComfyUI-GGUF (if using GGUF Flux)
5. **Build workflows** in the UI, export as API format JSON
6. **Write Python client** using the WebSocket API pattern shown above
7. **Test and iterate** on prompt engineering for different logo styles

### Prompt Engineering Tips for Logos

**Effective positive prompts:**
```
"minimalist vector logo for [business type], flat design, clean lines,
professional, simple background, [style keywords from LoRA trigger]"
```

**Effective negative prompts:**
```
"photorealistic, photograph, 3d render, blurry, noisy, complex background,
watermark, text artifacts, low quality, deformed"
```

**Style modifiers to try:**
- `vector`, `flat design`, `minimalist` -- for clean modern logos
- `emblem`, `badge`, `crest` -- for badge-style logos
- `mascot`, `cartoon`, `character` -- for mascot logos
- `wordmark`, `typography`, `lettering` -- for text-focused logos (use Flux)
- `monogram`, `lettermark` -- for initial-based logos
